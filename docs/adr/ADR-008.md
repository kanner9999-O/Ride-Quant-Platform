# ADR-008: Phân bổ ngôn ngữ theo layer

**Context:** Cần chọn ngôn ngữ triển khai đảm bảo [I-2 Decision Parity](../constitution/02-platform-invariants.md) (cùng logic quyết định giữa Research/Production) trong khi vẫn có concurrency/reliability tốt ở tầng I/O. Team có kinh nghiệm Python + Go/Rust. Latency yêu cầu: hàng chục-hàng trăm ms (không phải HFT sub-millisecond).

**Decision:**
- **Python** cho Feature Engineering, Strategy, Decision logic, Backtest Engine — vì I-2 yêu cầu cùng 1 hàm chạy ở cả Research và Production, và Python có ecosystem tốt nhất cho việc này (Polars/Pandas, TA-Lib, NumPy).
- **Go** cho Market Data Ingestion, Risk Gateway, Execution Engine — nơi cần concurrency (nhiều WS connection đồng thời), I/O, độ tin cậy.
- **Rust KHÔNG dùng ngay** — reserved cho critical path tương lai nếu latency profile thay đổi (hiện tại vài trăm ms không cần Rust). Giới thiệu Rust sau này là quyết định kiến trúc mới, cần ADR riêng (theo 4b — thay đổi ảnh hưởng nhiều module).

**Alternatives considered:**
- Rust cho toàn bộ edge ngay từ đầu — loại bỏ vì velocity phát triển chậm hơn Go, không tương xứng với latency requirement hiện tại.
- Python cho cả edge (Ingestion/Risk Gateway/Execution) — loại bỏ vì concurrency/reliability kém hơn Go cho I/O-heavy service.

**Concerns / Risks noted:** Risk Gateway viết Go (không phải Python) có nguy cơ tạo Parity gap nếu Backtest dùng một bản risk-check Python "rút gọn" riêng thay vì gọi qua đúng Risk Gateway Go thật — đây là lỗi kinh điển trong lịch sử hệ thống trading (risk model backtest lệch khỏi risk model production). Giảm thiểu: Parity Principle (I-2) không yêu cầu cùng NGÔN NGỮ, chỉ yêu cầu cùng CODE PATH/SERVICE INSTANCE — Backtest/Replay/Paper/Live đều phải gọi qua đúng 1 Risk Gateway service, không mock/simplify cho nghiên cứu (xem [03-engineering-principles.md](../constitution/03-engineering-principles.md)).

**Scale check:**
```yaml
scale_check:
  current_scale: "2-3 sàn, latency vài trăm ms"
  expected_scale:
    exchange: 20
    latency_requirement: "có thể xuống sub-ms nếu mở rộng market making"
  decision_still_valid: YES
  reason: >
    Phân bổ Python/Go theo layer (không theo tính năng cụ thể) scale tốt khi thêm
    exchange/strategy. Nếu latency requirement đổi sang sub-ms (ví dụ market making
    thật), cần ADR mới để đánh giá lại việc đưa Rust vào — đã được chừa chỗ.
  reason_if_no: ""
```

**Consequences:** Team cần duy trì kỷ luật ranh giới ngôn ngữ (không để logic nghiệp vụ lọt vào Go, không để Go/Rust bị dùng cho quyết định thay vì chỉ I/O). Risk Gateway (Go) phải được thiết kế để Backtest/Replay/Paper gọi được trực tiếp (không phải chỉ Live), tránh future engineer viết "phiên bản nhẹ" tách biệt.
